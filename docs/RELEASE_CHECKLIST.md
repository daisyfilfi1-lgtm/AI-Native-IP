# 双仓发布清单（Frontend: Netlify / Backend: Railway）

适用于当前项目的前后端分仓发布流程。目标：减少“只发一边”与“接口不对齐”问题。

## 一、发布前（需求与设计）

- 明确本次是否涉及接口变化：路径、请求字段、响应字段、错误码、鉴权。
- 若涉及数据库字段变更：必须先补 migration，并保证幂等（`IF NOT EXISTS`）。
- 若涉及第三方集成（飞书/百度等）：确认环境变量与配置入口是否齐备。

## 二、开发顺序（推荐）

1. 先改后端：接口、数据模型、迁移、文档。
2. 再改前端：类型定义、API 调用、交互与错误提示。
3. 本地联调：至少覆盖“成功 + 失败”两条路径。

## 三、提交规范（双仓）

- Backend 与 Frontend 分别提交，commit message 标注同一需求主题（便于追溯）。
- 不要在未验证接口的情况下只推前端。
- 不要在无 migration 的情况下推送依赖新列的后端代码。

## 四、推送与部署顺序

1. 推送 Backend 仓库（触发 Railway）。
2. 等后端部署成功并通过健康检查。
3. 推送 Frontend 仓库（触发 Netlify）。
4. 前后端都绿灯后再做线上验收。

## 五、上线后冒烟检查（最少 5 项）

- `GET /health` 返回正常。
- 新建 IP 成功（含基础字段与扩展字段）。
- 素材录入成功（URL 或本地上传链路至少测一条）。
- 飞书配置可保存、可读取、可同步。
- 百度网盘配置可保存、可同步（指定 `remote_path`）。

## 六、回滚策略

- 前端问题：优先回滚前端到上一个可用版本（不影响后端）。
- 后端问题：回滚后端版本；若已执行迁移，优先采用“向后兼容修复”而非直接删列。
- 紧急场景下，前端先临时降级相关入口（隐藏按钮/提示维护中）。

## 七、发布记录模板

每次发布后记录一条：

- 需求：`<一句话>`
- Backend commit：`<hash>`
- Frontend commit：`<hash>`
- Railway 部署：`成功/失败（链接）`
- Netlify 部署：`成功/失败（链接）`
- 冒烟结果：`通过/未通过（问题简述）`
