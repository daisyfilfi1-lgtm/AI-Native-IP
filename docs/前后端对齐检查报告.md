# 前后端对齐检查报告

基于当前后端数据库/接口与前端页面、类型、API 调用的对照结果。

---

## 一、已对齐部分

| 模块 | 后端 | 前端 | 说明 |
|------|------|------|------|
| **飞书同步** | GET/POST `/integrations/feishu/config`、GET `spaces`、POST `sync` | `api.getFeishuConfig/saveFeishuConfig/getFeishuSpaces/syncFeishu`，类型一致 | `GET /config` 返回**完整 app_id**（便于预填）；凭证保存、列空间、同步已对接 |
| **百度网盘** | GET/POST `/integrations/baidu/config`、POST `sync` | 需在记忆 Agent 页增加 Tab + `getBaiduConfig/saveBaiduConfig/syncBaidu` | 后端已就绪 |
| **IP 类型** | `ip_id, name, owner_user_id, status` | `IP`、`CreateIPRequest` 字段一致 | 创建单条、查询单条结构匹配 |
| **Ingest 响应** | `ingest_task_id, status` | `IngestResponse` | 一致 |
| **Ingest 状态** | `ingest_task_id, status, error?, created_assets` | `IngestStatus` | 一致 |
| **Memory 配置结构** | `MemoryFullConfig`（tag_config / memory_config） | `MemoryFullConfig`、`TagConfig`、`MemoryConfig`、`RetrievalConfig`、`UsageLimitsConfig` | 类型与后端 Pydantic 一致 |
| **FeishuSyncResult** | `synced, failed, errors` | 同名字段 | 一致 |

## 二、不对齐与缺失

### 1. IP 管理

| 项目 | 后端 | 前端 | 建议 |
|------|------|------|------|
| **IP 列表** | 仅有 GET `/ip/{ip_id}`，**无列表接口** | `/ip` 页使用 mock 数组 `mockIPs` | 后端增加 GET `/ip`（支持分页/筛选可选），前端改为调接口拉列表 |
| **创建 IP** | POST `/ip` 需要 `ip_id, name, owner_user_id, status?` | 新建 IP 弹窗无 `owner_user_id`，点击「创建」仅关弹窗，**未调 api.createIP** | 弹窗增加 `owner_user_id`（或默认值），提交时调用 `api.createIP`，成功后刷新列表或关闭 |

### 2. 素材录入（Memory Ingest）

| 项目 | 后端 | 前端 | 建议 |
|------|------|------|------|
| **请求体** | 必填：`ip_id, source_type`；`source_url` 与 `local_file_id` **至少其一**（先 `POST /memory/upload`） | 无 `ip_id`、无 URL/上传；未调 `api.ingestMemory` | 表单增加 IP、`source_url` 或本地上传链路、`api.uploadMemoryFile` + `api.ingestMemory` |
| **任务状态** | GET `/memory/ingest/{task_id}` 返回 `ingest_task_id, status, error?, created_assets` | `handleIngest` 仅本地 setInterval 模拟进度，未调 `api.getIngestStatus` | 提交后拿到 `ingest_task_id`，轮询 `api.getIngestStatus(task_id)`，用真实 status/created_assets 展示进度与结果 |

### 3. 语义检索

| 项目 | 后端 | 前端 | 建议 |
|------|------|------|------|
| **请求** | POST `/memory/retrieve`：`ip_id, query, filters?, top_k?` | 未传 `ip_id`，未调 `api.retrieveMemory`，用 mock 数据 | 增加 ip_id（URL 或选择），搜索时调用 `api.retrieveMemory` |
| **结果结构** | `results[]`: `asset_id, title?, content_snippet?, metadata, similarity` | mock 使用 `id, title, type, tags, usage, limit` | 结果列表按后端字段渲染（asset_id、title、content_snippet、similarity 等） |

### 4. Memory 高级配置（标签 + 检索/使用限制）

| 项目 | 后端 | 前端 | 建议 |
|------|------|------|------|
| **读取** | GET `/config/memory?ip_id=` 返回该 IP 的 tag_config + memory_config | 高级配置 Tab 未调 `api.getMemoryConfig`，表单项为静态默认值 | 进入页或切换 IP 时根据 `ip_id`（如 URL `?ip=`）调 `api.getMemoryConfig(ip_id)` 回填表单 |
| **保存** | POST `/config/memory` 收 `MemoryFullConfig`，至少其一，且含 `ip_id, config_id, updated_by` 等 | 未调 `api.saveMemoryConfig`，无「当前 IP」绑定 | 表单绑定 tag_config / memory_config 结构，保存时组好 payload 调 `api.saveMemoryConfig`，并传当前 ip_id（及 config_id、updated_by） |
| **config_id** | 后端 upsert 需要 config_id | 前端类型有，但若从未拉取则无 config_id | 首次保存可由后端生成或前端用 `memory_config_{ip_id}` 等约定；若 GET 返回则直接复用 |

### 5. 配置中心总览（/config）

| 项目 | 后端 | 前端 | 建议 |
|------|------|------|------|
| **按 IP 维度** | 配置均按 `ip_id` 读写 | 配置中心为全局 mock，未按 IP 拉取 | 若需「各 Agent 配置状态」来自真实数据，可先按 IP 拉 Memory 配置，再扩展其他 Agent 状态接口 |

### 6. 待办标签 / 更新标签

| 项目 | 后端 | 前端 | 建议 |
|------|------|------|------|
| **接口** | GET `/memory/pending-labels?ip_id=&limit=`；POST `/memory/labels/{asset_id}` body `{ ip_id, confirmed_labels }` | `api` 无 `getPendingLabels`、`updateLabels`；记忆 Agent 页无对应 UI | 若要做「待复核列表」与「确认标签」，需在前端加接口封装与页面调用 |

### 7. IP 响应字段

| 项目 | 后端 | 前端 | 说明 |
|------|------|------|------|
| **created_at / updated_at** | `IPResponse` 未返回 | `IP` 类型有 `created_at?, updated_at?`；列表 mock 有 | 若列表来自后端且需展示时间，可在 GET `/ip` 或 GET `/ip/{id}` 中返回这两个字段 |

## 三、数据库与前端类型对照（概要）

- **ip**：后端 `ip_id, name, owner_user_id, status, created_at, updated_at`；前端 `IP` 一致（时间可选）。
- **tag_config / memory_config**：后端 JSONB 与 Pydantic 与前端 `TagConfig` / `MemoryConfig` 结构一致；注意 `updated_at` 后端返回 isoformat 字符串。
- **ingest_tasks**：后端 `created_asset_ids`，接口返回为 `created_assets`，前端类型为 `created_assets`，一致。
- **integration_config**：仅后端使用，前端仅通过 Feishu 配置接口间接使用，无直接表结构对应。

---

## 四、建议修复优先级

1. **高**：IP 创建弹窗对接 `api.createIP`（并补全 `owner_user_id`）；素材录入对接 `api.ingestMemory` + `api.getIngestStatus`，表单补全 `ip_id`、`source_url`。
2. **高**：后端增加 GET `/ip` 列表接口；前端 IP 页用接口拉列表（或至少在创建后能反映新 IP）。
3. **中**：记忆 Agent 高级配置按 `ip_id`（如 URL 参数）拉取与保存，对接 `getMemoryConfig` / `saveMemoryConfig`。
4. **中**：语义检索页传 `ip_id` 并调用 `api.retrieveMemory`，结果按后端字段展示（当前后端占位返回空也可先接好）。
5. **低**：待办标签/更新标签的 API 封装与 UI；配置中心与真实配置状态对接。

完成上述修改后，管理后台与后端数据库、接口可对齐到当前 Phase 1 设计。

## 五、已完成的修复（本次检查后实施）

1. **后端**：新增 **GET /api/v1/ip**，返回所有 IP 列表（按创建时间倒序）。
2. **前端 IP 页**：使用 `api.listIPs()` 拉取列表；新建 IP 弹窗增加「负责人 ID」，提交时调用 `api.createIP`，成功后刷新列表。
3. **前端记忆 Agent - 素材录入**：增加「目标 IP ID」、按类型展示「URL 链接」；提交调用 `api.ingestMemory`，轮询 `api.getIngestStatus` 展示任务状态与完成结果。
4. **前端记忆 Agent - 高级配置**：增加「当前 IP ID」（与 URL `?ip=` 同步）；进入配置 Tab 时根据 ip_id 调用 `api.getMemoryConfig` 回填；表单绑定检索/使用限制，保存时调用 `api.saveMemoryConfig`（仅提交 memory_config，tag_config 可后续再接）。

## 六、后续优化（已一并完成）

1. **语义检索 Tab**：增加「目标 IP ID」；搜索时调用 `api.retrieveMemory({ ip_id, query, filters?, top_k })`，结果按后端字段 `asset_id / title / content_snippet / similarity / metadata` 展示；后端占位返回空时提示「接入向量库后将返回检索结果」。
2. **待办标签 / 更新标签**：前端封装 `api.getPendingLabels(ipId, limit)`、`api.updateLabels(assetId, { ip_id, confirmed_labels })`；在「标签管理」Tab 增加「待复核列表」区块，按 IP 拉取待复核项，展示 `asset_id / title / content_snippet / auto_labels`，每项提供「确认标签」：编辑 JSON 格式的 `confirmed_labels` 后提交，调用 `updateLabels` 并刷新列表。
3. **配置中心 /config**：增加「选择 IP」下拉（数据来自 `api.listIPs()`）；根据所选 IP 调用 `api.getMemoryConfig(ipId)`，将**记忆 Agent** 的配置状态与进度条改为接口结果（有 tag_config 或 memory_config 则已配置/配置中）；记忆 Agent 的「配置」链接带 `?ip=` 跳转到记忆页。
