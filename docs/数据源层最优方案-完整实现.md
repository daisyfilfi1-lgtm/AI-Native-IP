## 三、核心组件详解

### 3.1 抽象基类 `DataSource`

```python
class DataSource(ABC):
    """所有数据源必须实现此接口"""
    
    @abstractmethod
    async def fetch(self, ip_profile, limit) -> List[TopicData]:
        """获取话题数据"""
        pass
    
    @abstractmethod  
    def is_available(self) -> bool:
        """检查是否可用"""
        pass
    
    # 内置健康状态管理
    health: DataSourceHealth
```

**设计意义**：统一接口，任何数据源都可即插即用

### 3.2 数据源管理器 `DataSourceManager`

**职责**：
- 管理所有数据源生命周期
- 多源融合策略（hybrid/priority/parallel）
- 自动故障转移
- 数据去重

**使用示例**：
```python
manager = get_datasource_manager()

# 一键获取话题（自动多源融合）
topics = await manager.fetch_topics(ip_profile, limit=12)

# 查看数据源健康状态
health = await manager.health_check()
```

### 3.3 本地缓存层 `TopicCache`

**设计**：
- 二级缓存：内存 + 磁盘
- TTL自动过期
- API失败时读取过期缓存（降级）

**效果**：
- TIKHUB失效时，仍可返回30分钟内的缓存数据
- 缓存完全解耦外部依赖

### 3.4 内置数据源 `BuiltinDataSource`

**设计**：
- 完全零外部依赖
- 按4-3-2-1内容矩阵组织
- 支持YAML配置热加载
- 优先级最低(P3)，作为最终兜底

**数据示例**：
```yaml
xiaomin1:
  money:
    - title: "2000块启动资金，我是如何做到月入3万的"
      score: 4.95
  emotion:
    - title: "32岁离婚带俩娃，我是怎么靠自己走出低谷的"
      score: 4.94
```

## 四、多源融合策略

### 4.1 Hybrid策略（推荐）

**逻辑**：
1. 按优先级尝试实时数据源
2. 实时数据不足，用内置库补充内容矩阵
3. 自动去重

**代码**：
```python
async def _fetch_hybrid(ip_profile, limit):
    result = []
    
    # 获取实时数据
    for source in sorted_sources:
        topics = await source.fetch(ip_profile, limit)
        result.extend(topics)
        if len(result) >= limit:
            break
    
    # 补充内置库
    if len(result) < limit:
        builtin_topics = await builtin.fetch(ip_profile, needed)
        result = fill_content_matrix(result, builtin_topics, limit)
    
    return result[:limit]
```

### 4.2 故障处理机制

```
Level 1: 数据源内部
   缓存 → API → 过期缓存

Level 2: 管理器层
   多源优先级队列 → 自动跳过不可用源

Level 3: 全局兜底
   内置数据源 (始终可用)
```
