# 数据源层最优架构设计

## 一、架构目标

### 核心诉求
1. **零依赖风险** - 任何单一数据源失效，系统仍能正常工作
2. **高质量数据** - 优先使用精准数据，兜底使用内置数据
3. **可扩展性** - 新增数据源只需实现统一接口
4. **可观测性** - 完整的健康检查和监控

### 2.2 核心组件

#### 1. 抽象基类 `DataSource`
```python
class DataSource(ABC):
    @abstractmethod
    async def fetch(self, ip_profile, limit) -> List[TopicData]
    
    @abstractmethod
    def is_available(self) -> bool
    
    # 健康状态管理
    health: DataSourceHealth
```

#### 2. 数据源管理器 `DataSourceManager`
- **单例模式** - 全局唯一实例
- **多源融合** - 支持 hybrid/priority/parallel 三种策略
- **健康检查** - 自动故障转移
- **数据去重** - 基于标题的智能去重

#### 3. 本地缓存层 `TopicCache`
- **二级缓存** - 内存 + 磁盘
- **TTL管理** - 自动过期
- **降级读取** - API失败时读取过期缓存

#### 4. 内置数据源 `BuiltinDataSource`
- **零依赖** - 不依赖任何外部服务
- **热更新** - 支持YAML配置热加载
- **矩阵分布** - 内置4-3-2-1内容矩阵

## 三、多源融合策略

### 3.1 Hybrid策略（推荐）

```python
async def _fetch_hybrid(ip_profile, limit):
    result = []
    
    # 1. 按优先级获取实时数据
    for source in sorted_sources:
        topics = await source.fetch(ip_profile, limit)
        result.extend(topics)
        if len(result) >= limit:
            break
    
    # 2. 实时数据不足，用内置库补充内容矩阵
    if len(result) < limit:
        needed = limit - len(result)
        builtin_topics = await builtin.fetch(ip_profile, needed)
        result = fill_content_matrix(result, builtin_topics, limit)
    
    return result[:limit]
```

**适用场景**：生产环境默认策略

### 3.2 Priority策略

```python
async def _fetch_priority(ip_profile, limit):
    for source in sorted_sources:
        topics = await source.fetch(ip_profile, limit)
        if topics:
            return topics[:limit]
    return []
```

**适用场景**：追求响应速度，接受单一数据源

### 3.3 Parallel策略

```python
async def _fetch_parallel(ip_profile, limit):
    tasks = [source.fetch(ip_profile, limit) for source in sources]
    results = await asyncio.gather(*tasks)
    
    all_topics = []
    for topics in results:
        all_topics.extend(topics)
    
    return deduplicate(all_topics)[:limit]
```

**适用场景**：需要最大化数据覆盖面

## 四、故障处理机制

### 4.1 三级故障处理

```
Level 1: 数据源内部处理
├── 缓存读取（优先）
├── API调用（实时）
└── 过期缓存兜底（API失败）

Level 2: 管理器层处理
├── 多源优先级队列
├── 自动跳过不可用源
└── 混合策略补充

Level 3: 全局兜底
├── 内置数据源（始终可用）
└── 4-3-2-1矩阵保证
```

### 4.2 健康检查

```python
@dataclass
class DataSourceHealth:
    status: DataSourceStatus  # HEALTHY/DEGRADED/UNAVAILABLE
    success_rate: float       # 最近1小时成功率
    response_time_ms: float   # 平均响应时间
    error_count: int          # 连续错误次数
```

**状态转换**：
- HEALTHY → DEGRADED: 连续2次错误
- DEGRADED → UNAVAILABLE: 连续5次错误
- UNAVAILABLE → HEALTHY: 成功率>80%

## 五、数据源优先级

| 优先级 | 数据源 | 类型 | 延迟 | 质量 | 可用性 |
|-------|--------|------|------|------|--------|
| P0 | 竞品监控 | 实时 | 中 | 极高 | 中 |
| P1 | 关键词搜索 | 实时 | 中 | 高 | 中 |
| P1 | 小红书话题 | 实时 | 中 | 高 | 中 |
| P2 | TIKHUB热榜 | 实时 | 低 | 中 | 低 |
| P3 | 内置爆款库 | 静态 | 极低 | 高 | 极高(100%) |

**设计原则**：
1. 优先使用精准数据（竞品、搜索）
2. 其次使用泛数据（热榜）
3. 始终保留兜底数据（内置库）

### 6.1 实现统一接口

```python
class NewDataSource(DataSource):
    def __init__(self, config: DataSourceConfig):
        super().__init__(config)
    
    def is_available(self) -> bool:
        # 检查是否配置/可用
        return True
    
    async def fetch(self, ip_profile: Dict, limit: int) -> List[TopicData]:
        # 1. 检查缓存
        cached = self.cache.get(...)
        if cached:
            return cached
        
        # 2. 调用API
        data = await self._call_api(...)
        
        # 3. 转换为标准格式
        topics = [TopicData(...) for d in data]
        
        # 4. 写入缓存
        self.cache.set(..., topics)
        
        return topics
```

## 七、与业务层集成

### 7.1 推荐服务调用

```python
# backend/app/services/topic_recommendation_v3.py

from app.services.datasource import get_datasource_manager

class TopicRecommendationServiceV3:
    async def recommend_topics(self, db, ip_id: str, limit: int = 12):
        # 1. 获取IP画像
        ip_profile = self._get_ip_profile(db, ip_id)
        
        # 2. 调用数据源层（无需关心具体数据源）
        manager = get_datasource_manager()
        topics = await manager.fetch_topics(ip_profile, limit, strategy="hybrid")
        
        # 3. 业务层处理（匹配、评分、排序）
        scored_topics = self._score_and_rank(topics, ip_profile)
        
        return scored_topics
```

## 八、优势总结

### 相比V1.0

| 维度 | V1.0 | V2.0 (本架构) |
|------|------|--------------|
| **耦合度** | 强耦合TIKHUB | 完全解耦，可插拔 |
| **可用性** | TIKHUB失效即不可用 | 内置库兜底，100%可用 |
| **扩展性** | 修改困难 | 新增数据源只需实现接口 |
| **可观测** | 无健康检查 | 完整健康状态和监控 |
| **数据质量** | 依赖外部 | 多源融合+矩阵分配 |
| **响应速度** | 依赖网络 | 缓存优先，极速响应 |

### 关键创新

1. **抽象层设计** - 统一接口，数据源即插即用
2. **多级缓存** - 内存+磁盘，API失败仍可服务
3. **智能融合** - 多源数据去重+内容矩阵填充
4. **健康检查** - 自动故障转移，无人工干预
5. **零依赖兜底** - 内置库保证100%可用性
