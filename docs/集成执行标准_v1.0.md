# 集成执行标准 v1.0

适用范围：本次 IP 工厂升级开发（知识库、Agent 编排、生成链路、审核与发布链路）。  
目标：确保系统集成可落地、可观测、可回滚，避免出现长期 `PROCESSING`、状态不一致、线上不可定位问题。

---

## 1. 核心原则（必须遵守）

1. 单一编排中心：同一任务只能由一个系统推进状态（禁止多系统同时写状态）。
2. 单一事实源：同一类核心数据只能有一个主系统（如向量检索主线只能一条）。
3. 显式状态机：所有异步任务必须按统一状态流转。
4. 幂等优先：所有重试必须安全，不得产生重复资产或脏状态。
5. 可观测先行：未接入日志、指标、追踪前，不允许上线核心集成。
6. 失败可恢复：每个阶段必须定义超时、重试、补偿与人工介入机制。

---

## 2. 统一状态机标准

所有异步任务（ingest/sync/generate/render）必须使用如下状态：

- `QUEUED`
- `PROCESSING`
- `COMPLETED`
- `FAILED`
- `CANCELLED`
- `TIMEOUT`

### 2.1 状态流转规则

- 允许流转：
  - `QUEUED -> PROCESSING`
  - `PROCESSING -> COMPLETED|FAILED|TIMEOUT|CANCELLED`
- 禁止流转：
  - `COMPLETED/FAILED/TIMEOUT/CANCELLED -> PROCESSING`
- 任何异常退出必须落到终态（`FAILED` 或 `TIMEOUT`），禁止悬空状态。

### 2.2 心跳与卡死判定

- `PROCESSING` 任务必须每 30~60 秒刷新 `updated_at` 或 `last_heartbeat`。
- 超过阈值（建议 3~5 分钟）无心跳，系统自动标记 `TIMEOUT` 并记录原因。

---

## 3. 接口与契约标准

### 3.1 API 契约

- 统一版本前缀：`/api/v1`
- 所有响应必须包含可追踪字段：
  - `trace_id`
  - `task_id`（异步接口）
- 错误响应统一结构：
  - `code`（稳定机器码）
  - `message`（用户可读）
  - `detail`（调试信息，可选）
  - `trace_id`

### 3.2 兼容性要求

- 生产接口变更必须保持向后兼容至少一个小版本周期。
- 不允许在未公告的情况下修改必填字段、语义或状态枚举。

---

## 4. 数据与幂等标准

### 4.1 关键字段统一

所有任务表至少包含：

- `task_id`（唯一）
- `ip_id`
- `status`
- `error_message`
- `retry_count`
- `created_at`
- `updated_at`

### 4.2 幂等键规范

- 写入型任务使用 `idempotency_key = task_id + stage`。
- 对外回调或重复请求必须先查幂等键，避免重复写资产。
- 所有“重试”逻辑必须可重复执行且结果一致。

---

## 5. 任务执行与资源隔离标准

1. Web 进程与 Worker 进程必须隔离部署（禁止长任务占用 Web 请求线程）。
2. 外部依赖调用（LLM、Embedding、对象存储）必须设置：
   - 连接超时
   - 读取超时
   - 最大重试次数
3. 禁止无限阻塞调用；必须有可终止策略（超时/熔断/降级）。
4. 长任务阶段性提交与落盘，避免整段失败导致状态不可恢复。

---

## 6. 可观测性标准（上线前必备）

### 6.1 日志

结构化日志最小字段：

- `timestamp`
- `level`
- `trace_id`
- `task_id`
- `ip_id`
- `stage`
- `status`
- `latency_ms`
- `provider`（外部依赖时）
- `error_code` / `error_message`

### 6.2 指标

必须有以下指标面板：

- 任务成功率（按任务类型）
- `PROCESSING` 超时/卡死率
- 各阶段 P95/P99 耗时
- 外部依赖错误率（按 provider）
- 重试次数分布
- 死信/补偿任务数量

---

## 7. 质量门禁（Gate）

### Gate 0：设计冻结

- 已确定单一编排中心
- 已确定单一向量主线
- 状态机与补偿策略已评审通过

### Gate 1：契约测试

- 核心接口 schema 测试通过
- 错误码、状态码、终态语义一致

### Gate 2：异步可靠性

- 连续 500 个任务无永久 `PROCESSING`
- 超时任务可自动进入 `TIMEOUT/FAILED`
- 重试不产生重复资产

### Gate 3：稳定性压测

- 在目标并发下核心链路可用
- P95、错误率达标
- 外部依赖故障注入测试通过

### Gate 4：灰度发布

- 10% 流量灰度 24 小时
- 关键指标无恶化
- 一键回滚脚本验证通过

---

## 8. 发布与回滚标准

1. 所有发布必须带版本号与变更清单。
2. 生产发布必须具备：
   - 回滚步骤
   - 数据兼容策略
   - 风险项与观察指标
3. 当出现以下任一情况，立即回滚：
   - `PROCESSING` 卡死率超阈值
   - 核心接口 5xx 持续升高
   - 关键链路 P95 显著劣化

---

## 9. 故障演练（每周至少一次）

至少覆盖以下故障场景：

1. LLM 超时/失败（验证降级与重试）
2. 向量库不可用（验证回退策略）
3. 对象存储抖动（验证超时与失败落态）
4. Worker 重启（验证任务恢复与幂等）

演练输出必须包含：

- 触发时间
- 影响范围
- 检测时延
- 恢复时长
- 防复发动作

## 11. 角色与责任（RACI 简版）

- 技术负责人：架构冻结、门禁放行、回滚决策
- 后端负责人：状态机、任务执行、外部依赖治理
- 前端负责人：状态展示一致性、错误可读性
- QA/测试：契约测试、集成回归、故障演练
- 运维/平台：部署、监控、告警、容量管理
